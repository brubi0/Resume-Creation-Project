  <action>
Create a new file `system/profile-generation-guide.md` that serves as the complete protocol for generating a role profile. The file must cover:

**1. When this guide applies**
This guide runs when workflow.md Phase 0 finds no matching profile for the target role.

**2. Input collection**
Before researching, confirm you have: (a) target role/title, (b) industry or domain, (c) any must-have tools/certs mentioned by the candidate. These come from the Phase 0 intake questions already answered — do not re-ask.

**3. Research step (per D-01, D-02)**
Search for current job postings using the following queries (adjust role/industry):
- `"[target role]" job posting requirements site:linkedin.com OR site:indeed.com`
- `"[target role]" "required skills" OR "qualifications" -resume`
- `[target role] [industry] skills responsibilities 2024 OR 2025`

Analyze 3-5 postings. For each posting, extract:
- Required technical skills and tools
- Required soft skills or domain knowledge
- Certifications or qualifications listed
- Common job responsibilities (signals what skills matter most)

**4. Fallback when search is unavailable or returns fewer than 3 relevant postings (per D-03)**
If web search is unavailable or fewer than 3 relevant postings are found:
- Use Claude's training knowledge to construct the profile
- Add a `**Source:** Knowledge-based (no web search conducted)` line in the profile header, immediately after Target Roles
- Inform the user: "This profile was generated from training knowledge, not live job postings. Please review skill categories and proficiency definitions before proceeding."

**5. Paste-URL or paste-text alternative (per D-04)**
If the user has provided job posting URLs or raw posting text directly:
- Use those postings as your 3-5 sources instead of searching
- No search needed; proceed directly to extraction

**6. Extraction and grouping**
After gathering postings, identify 8-20 distinct skill categories. Group them into 2-4 logical groups (e.g., "Core Skills", "Technical Tools", "Soft Skills / Domain Knowledge"). For roles with a software product at the center (like NetSuite), name the core group after the product. For generalist roles, use domain-area groupings.

Each category must have:
- A clear, role-appropriate name (2-5 words)
- A one-paragraph description (2-4 sentences) explaining what knowledge/tools/tasks this category covers — this paragraph is used by Claude during the skills matrix discovery interview, so it must be specific enough to guide informed questions

**7. Proficiency level definitions**
Use the standard four-level system unless the role clearly needs different terminology:
- Expert (E) — Dark Blue (#3d5a80): Deep expertise, can teach others, handles complex scenarios
- Advanced (A) — Gold (#e9c46a): Strong working knowledge, independent problem-solving
- Intermediate (I) — Green (#52b788): Functional knowledge, can handle routine tasks
- Novice (N) — Coral (#e76f51): Basic familiarity, still learning

For highly specialized roles (e.g., surgeon, pilot), note adapted terminology while keeping the same 4 levels and hex codes.

**8. Output format**
Save the generated profile to `profiles/[role_slug].md` where role_slug is lowercase, underscores for spaces (e.g., `financial_controller.md`, `frontend_engineer.md`). The file must use the exact structure from `profiles/netsuite_administrator.md`:

```
# Profile: [Role Name]

**Industry:** [Industry]
**Target Roles:** [Role 1], [Role 2], [Role 3...]
<!-- Add this line ONLY for knowledge-based profiles: -->
**Source:** Knowledge-based (not research-backed — validate before use)

---

## Proficiency Levels

| Level | Code | Color | Hex | Meaning |
...

---

## Skills Categories

### [Group Name]

**[Category Name]**
[Description paragraph]
...
```

**9. Confirm with user**
After generating and saving the profile, display a summary:
- Profile saved to: `profiles/[role_slug].md`
- Number of skill categories: N across M groups
- Source: [web search / knowledge-based / user-provided postings]
- Ask: "Does this profile look right, or should I adjust any categories before we continue?"

Do not proceed to Phase 1 until the user confirms or requests changes.
